# 🧠 认知对齐 CogAlign

> 不看代码写得对不对，看写代码的人脑子里想的是不是一回事。

检查不同模块对同一个概念的理解是否一致。扫描类型定义、枚举、命名、注释，检测 7 类认知冲突，生成结构化认知地图报告。

---

## 为什么需要它？

代码审查能发现 bug。测试能发现逻辑错误。Linter 能发现风格问题。

但没有人告诉你——

```
auth 模块:    User = { id, email, passwordHash }    → "账户凭证"
profile 模块: User = { id, nickname, avatar }       → "个人资料"  
payment 模块: User = { id, credit, vipLevel }       → "消费主体"
```

三个模块。同一个 `User`。三个完全不同的含义。TypeScript 编译过了，测试全绿。但写这三个模块的人——对 "User 是什么" 的理解从未对齐。

**这是认知债务。总有一天会变成 bug。**

---

## 检测能力

| # | 冲突类型 | 例子 |
|---|---------|------|
| 1 | 命名冲突 | `user` 在 auth 指账户、在 profile 指资料、在 payment 指消费主体 |
| 2 | 状态空间不一致 | A 的枚举有 5 个值，B 只认识 3 个 |
| 3 | 隐含假设冲突 | A 假设 id 不可变，B 修改了 id |
| 4 | 生命周期歧义 | 3 个 controller 都能删 User，级联规则不同 |
| 5 | 边界条件差异 | A 认为空数组合法，B 视为错误 |
| 6 | 默认值战争 | 同一字段在不同模块默认值不同 |
| 7 | 验证逻辑重复 | email 验证在 3 处实现，正则各不相同 |

## 使用方式

```
/cogalign                                    # 分析当前工作目录
认知对齐                                      # 最简触发
检查这个项目的认知对齐                          # 自然语言
扫描 src/ 下的认知冲突                         # 聚焦子目录
order 和 payment 对 User 的理解一致吗           # 指定概念对比
这个项目的认知债务有哪些                        # 侧重债务清单
```

## 输出报告

7 章结构化 Markdown 报告，每个发现标注源码位置和严重级别：

```
🧠 认知地图 · 项目名称

一、领域概念清单          ← 提取的核心概念，一致性评级（✅/❓/🔴）
二、模块认知矩阵          ← 每个模块对概念的"理解"对比表
三、认知冲突清单          ← 🔴严重 / 🟡需关注 / 🟢低风险
四、共识边界              ← 团队共识 + 模糊地带 + 潜在冲突
五、认知债务清单          ← 现在没问题、将来会炸的隐患
六、心智模型推断图        ← ASCII 实体关系图 + 冲突标注
七、行动建议              ← 按优先级排序，含具体修复方向
```

## 与其他工具的配合

| 工具 | 做什么 | 使用时机 |
|------|--------|---------|
| **CogAlign** | 认知一致性、隐含假设 | 接手老项目、重构前、新人入职 |
| code-review | bug、性能、简化 | 每次改代码前 |
| nuwa-skill | 蒸馏人的思维 → AI 角色 | 创建角色 prompt |

- **前置 code-review**：先认知对齐，再代码审查
- **重构前必跑**：避免在认知不一致的基础上重构
- **新人 onboarding**：接手老项目时第一件事
