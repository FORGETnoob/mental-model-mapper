# 认知对齐 CogAlign

检查同一个概念在不同模块中是否被一致地理解和使用。
不检查语法错误。只检查**认知一致性**——写代码的人脑子里想的是不是一回事。

## 工作流

### 第一步：扫描与概念提取

从代码库的以下信号中提取 10-25 个核心领域概念：

1. **类型定义** — interface/type/struct/class 的字段共同定义"什么是 X"
2. **枚举值** — 每个枚举成员是作者对状态空间的一次切割
3. **变量命名** — 同一词根在不同文件指不同事物，是认知偏差的主要信号
4. **函数签名** — 参数名和返回类型隐含输入/输出假设
5. **注释** — "为什么这样做"比"做了什么"更反映认知；`// TODO` 和 `// HACK` 标记认知债务
6. **条件分支** — `if status === 'pending'` 隐含了对该状态行为的假设
7. **数据库 schema** — 表结构、外键、约束是领域关系的权威编码

### 第二步：模块认知映射

按目录/文件分组为模块。对每个核心概念，逐模块记录其"理解方式"：

| 模块 | 认知立场 | 支持的状态/值 | 隐含假设 | 证据 |
|------|---------|--------------|---------|------|
| [模块A] | [一句话] | [枚举] | [假设] | [file:line] |

同时记录：实体还是事件？可变还是不可变？识别的状态是全部枚举还是子集？

### 第三步：认知冲突检测

7 个维度横向比较：

1. **命名冲突** — 同一词根在不同模块指不同事物
2. **状态空间不一致** — 模块 A 的状态机包含模块 B 不认识的过渡
3. **隐含假设冲突** — 模块 A 假设 id 不可变，模块 B 修改了 id
4. **生命周期歧义** — 谁创建、谁删除、谁拥有所有权不明确
5. **边界条件差异** — 模块 A 认为空数组合法，模块 B 视为错误
6. **默认值战争** — 不同模块对同一字段设不同默认值
7. **验证逻辑重复** — 同一验证规则在多处实现但规则不同

每个冲突评级：🔴 严重 / 🟡 需关注 / 🟢 低风险

### 第四步：生成认知地图

输出结构化 Markdown 报告，含 7 个章节：
领域概念清单、模块认知矩阵、认知冲突清单、共识边界、认知债务清单、心智模型推断图、行动建议。

## 执行规则

1. 未指定目录时先询问。代码库 >500 文件时建议聚焦核心模块
2. 每个发现附带 file:line 引用
3. 只从代码推断，不猜测"作者应该想的是……"
4. 区分缺失（职责分离）与冲突（理解不同）
5. 无发现的章节保留标题注明"未发现"

## 使用方式

```
/cogalign                                    # 分析当前目录
检查这个项目的认知对齐                          # 自然语言
扫描 src/ 的认知冲突                           # 聚焦子目录
order 和 payment 模块对 User 的理解一致吗       # 指定概念对比
```

## 与其他 skill 的关系

- **vs code-review**：你检查认知对齐，它检查代码质量。互补不重叠
- **vs nuwa-skill**：互为镜像——它蒸馏人的思维，你蒸馏代码的思维
- **前置于重构**：重构前先跑一次，建立认知基线
